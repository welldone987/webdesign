# 项目规范

本文档是 AI 开发执行的主要依据，合并了项目计划与开发规范。`AGENTS.md` 只作为入口提示，具体规则以本文档为准。

## 第一部分：项目计划

### 1. 项目目标

建设一组个人静态网站，后续通过不同子域名区分访问入口：

```text
主页网站
摄影集网站
简历网站
```

当前阶段暂不开发主页网站和简历网站，主要开发摄影集网站。整体结构需要为后续三网站并行开发、独立部署、独立绑定子域名预留空间。

网站采用静态发布方式，不做复杂后端、不做用户系统、不做数据库管理。核心目标是页面简洁、加载快速、视觉质感好、后续维护方便。

### 2. 已确定方案

- 网站类型：静态个人网站群
- 当前开发重点：摄影集网站
- 后续网站方向：主页网站 + 摄影集网站 + 简历网站
- 前端技术栈：React + Vite + TypeScript + Tailwind CSS
- 动效方案：Framer Motion
- 项目结构：按多网站结构组织，优先考虑 monorepo / apps 目录
- 部署平台：Cloudflare Pages
- 代码管理：Git + GitHub
- 发布方式：GitHub 仓库连接 Cloudflare Pages，当前先部署摄影集网站
- CI/CD：优先使用 Cloudflare Pages Git 集成，不单独配置 GitHub Actions
- 代码质量：第一版只使用 TypeScript 做基础类型约束
- 包管理：第一版使用 npm，并提交 `package-lock.json`
- Node.js 版本：使用当前 LTS，并通过 `.nvmrc` 或 `.node-version` 固定版本，保证本地和 Cloudflare Pages 构建一致
- 访问统计：不收集任何访问数据
- 域名策略：第一版先使用 Cloudflare Pages 默认域名；后续购买个人域名后，用子域名区分三个网站
- 图片管理：使用 Markdown / JSON 管理摄影作品元信息
- 图片优化：优先在提交前或独立脚本中完成压缩；如果图片放在 `public` 目录，不依赖 Vite 默认构建流程自动压缩
- 待定事项：图片文件放在 `public` 本地目录，还是使用外部图床 / 对象存储

### 3. 技术栈说明

React 用于组件化开发页面。当前主要用于摄影集网站的画廊、作品网格、全屏查看、分类筛选等模块。后续也可复用到主页网站和简历网站。

Vite 用于本地开发与生产打包。开发时启动速度快，配置简单，适合静态网站。

TypeScript 用于给项目数据和组件参数增加类型约束，降低后续维护时字段混乱和修改出错的概率。

Tailwind CSS 用于快速编写响应式样式。适合摄影集网站这类需要精细排版、移动端适配和统一视觉风格的网站。

Framer Motion 用于实现页面进入、区块过渡、图片悬停、摄影作品切换等流畅动效。动效应保持克制，服务于阅读和作品展示，不喧宾夺主。

Cloudflare Pages 用于托管静态网站。优点是免费额度充足、全球 CDN、支持自动构建、支持自定义域名和 HTTPS。

### 4. 网站结构规划

后续子域名结构：

```text
www.example.com       主页网站
photo.example.com     摄影集网站
resume.example.com    简历网站
```

第一版暂不绑定个人域名，先使用 Cloudflare Pages 默认域名。后续购买并接入个人域名后，再映射到不同子域名。

当前优先开发摄影集网站。核心内容包括：

- 摄影作品首页
- 摄影作品分类
- 作品网格 / 瀑布流 / 横向画廊等多方式排版
- 单张图片放大查看
- 图片标题、拍摄地点、年份等基础信息
- 简短摄影介绍或 About 区域
- 联系方式

摄影作品分类示例：

- 人像
- 街拍
- 风景
- 旅行
- 黑白

展示排版采用多方式结合，暂不指定单一布局。可按具体作品内容组合使用网格、瀑布流、横向画廊、全屏浏览等形式。

主页网站后续作为统一入口开发，可能包含个人身份与定位、跳转到摄影集网站、跳转到简历网站、代表性视觉封面和联系方式。

简历网站后续单独开发，不和摄影集网站混在一个页面中，可能包含个人简介、技能栈、教育经历、工作 / 实习 / 项目经历、代表项目展示、PDF 简历下载和联系方式。

### 6. 开发步骤

第一阶段是内容整理：

1. 筛选摄影作品，先控制在 20 到 50 张以内。
2. 为摄影作品整理分类、标题、地点、年份等元信息。
3. 准备摄影集网站封面图。
4. 准备简短摄影介绍和联系方式。
5. 暂不整理主页和简历网站内容，后续单独处理。

第二阶段是项目初始化：

1. 创建多网站项目根目录。
2. 在 `apps/photography` 中创建 React + Vite + TypeScript 项目。
3. 安装并配置 Tailwind CSS。
4. 安装并配置 Framer Motion。
5. 建立摄影集网站基础目录结构。
6. 配置 TypeScript。
7. 固定 Node.js 版本，并提交 npm lockfile。
8. 为 `apps/home` 和 `apps/resume` 预留占位目录。
9. 初始化 Git 仓库并推送到 GitHub。

第三阶段是摄影集网站开发：

1. 开发摄影集网站整体布局。
2. 开发摄影集首页 Hero 区域。
3. 开发摄影作品分类导航。
4. 开发作品网格、瀑布流或其他组合展示模块。
5. 开发单张图片放大查看模块。
6. 开发摄影介绍、联系方式和页脚。
7. 完成桌面端和移动端响应式适配。

第四阶段是视觉与动效：

1. 确定整体视觉风格：简洁、专业、适度艺术感。
2. 使用 Framer Motion 给图片网格增加 hover 效果。
3. 使用 Framer Motion 给页面区块增加轻量进入动画。
4. 给摄影作品增加放大查看效果。
5. 避免过多动画影响阅读和加载性能。

第五阶段是图片优化：

1. 将大图压缩到适合网页展示的尺寸。
2. 优先使用 WebP 或 AVIF 格式。
3. 为图片设置固定宽高或 aspect-ratio，避免页面跳动。
4. 使用 lazy loading 延迟加载非首屏图片。
5. 不上传未处理的超大原图。
6. 如果图片放在 `public/images/photography/`，需通过提交前处理或 `npm run optimize:images` 等独立脚本完成压缩。
7. 如果图片放在 `src/assets/`，可再考虑接入 Vite 图片处理插件，但第一版不强制。
8. 使用 Markdown / JSON 记录摄影作品的分类、标题、地点、年份、图片路径等元信息。
9. 每张图片元信息至少包含 `src`、`alt`、`width`、`height`、`category`、`title`、`year`，可选包含 `location`、`featured`、`order`、`slug`。

### 7. 部署与 CI/CD

当前阶段部署摄影集网站：

1. 在 GitHub 创建代码仓库。
2. 将本地代码推送到 GitHub。
3. 在 Cloudflare Pages 创建摄影集网站项目。
4. 连接 GitHub 仓库。
5. 设置 Root directory 为 `apps/photography`。
6. 设置构建命令为 `npm run build`。
7. 设置输出目录为 `dist`。
8. 完成摄影集网站首次部署。
9. 测试线上访问效果。
10. 第一版先使用 Cloudflare Pages 默认域名。

后续三个网站可使用同一个 GitHub 仓库，但在 Cloudflare Pages 中创建三个独立项目：

```text
Cloudflare Pages Project 1 -> apps/home        -> www.example.com
Cloudflare Pages Project 2 -> apps/photography -> photo.example.com
Cloudflare Pages Project 3 -> apps/resume      -> resume.example.com
```

每个 Cloudflare Pages 项目设置不同的 Root directory、构建命令和输出目录。这样三个网站可以独立构建、独立部署、独立绑定域名。

当前不单独配置 GitHub Actions。后续如果三网站都加入仓库，可以考虑再增加 GitHub Actions 做统一检查，例如：

```text
typecheck
build:photography
build:home
build:resume
image optimize check
```

Cloudflare Pages 配置要点：

- Framework preset：Vite
- Root directory：`apps/photography`
- Build command：`npm run build`
- Build output directory：`dist`
- Node.js version：使用当前 LTS 版本，并与仓库中的 `.nvmrc` 或 `.node-version` 保持一致
- 部署方式：Cloudflare Pages Git 集成
- 不单独配置 GitHub Actions，除非后续需要更复杂的检查流程

如果后续绑定自定义域名，需要在 Cloudflare 中配置 DNS 解析。

### 8. 维护方式

摄影集网站后续更新内容时，优先修改数据文件：

```text
apps/photography/src/data/photos.json
apps/photography/src/content/galleries/
```

推荐原则：

- 修改文字内容时尽量不动组件结构。
- 新增摄影作品时优先更新图片文件和 `photos.json` / Markdown 内容文件。
- 如果采用 JSON / Markdown 管理摄影作品，则新增作品时优先更新对应 JSON / Markdown 文件。
- 页面样式修改集中在对应组件内完成。
- 第一版不接入访问统计，避免额外数据收集。
- 主页网站和简历网站后续作为独立应用维护，避免三个站点内容互相耦合。

### 9. 注意事项

- 不要在网站中暴露手机号、身份证、密钥、服务器密码等敏感信息。
- 摄影作品不要上传未经压缩的原图。
- 移动端体验必须重点检查。
- 所有摄影作品必须提供有意义的 `alt` 文本，不使用空泛描述。
- 图片放大查看器必须支持键盘关闭、焦点管理和移动端触控操作。
- 页面动效需尊重用户的 reduced-motion 偏好。
- 页面需要包含基础 SEO 信息，例如标题、描述、favicon 和 Open Graph 图片。
- 摄影作品集应让图片成为主角，不要让动画和装饰抢走注意力。
- 构建时图片压缩要保留足够画质，避免摄影作品被过度压缩。
- 若使用外部图床或对象存储，需要确认费用、访问速度、长期可控性和图片备份方案。
- 不接入访问统计时，无法直接知道访问量、来源和用户行为，这是主动选择的隐私与简化取舍。
- 三网站拆分后，需要注意共享依赖版本和构建配置的一致性，避免后续维护成本上升。

### 10. 后续可扩展方向

- 绑定个人域名。
- 使用子域名区分内容，例如 `www`、`photo`、`resume`。
- 增加摄影作品详情页。
- 增加 Markdown / MDX 内容管理。
- 初始化主页网站。
- 初始化简历网站。
- 提取三网站共享配置或共享 UI 组件。
- 如果后续需要更严格流程，再增加 GitHub Actions 做 lint、typecheck、build 检查。

### 11. 待定决策

图片存放位置待定：图片放在 `public` 本地目录，还是使用外部图床 / 对象存储。

可选方案：

- `public` 本地目录：最简单，适合第一版和图片数量不大的情况。
- 外部图床 / 对象存储：适合大量图片、高清图、后续独立管理图片资源。

需要根据摄影作品数量、单张图片大小、是否需要长期高清展示来决定。

### 12. 当前推荐路线

先完成一个可上线的第一版：

```text
摄影集网站独立上线
```

第一版上线后，再根据实际使用反馈逐步增强视觉、动效和内容管理方式。主页网站和简历网站后续再作为独立站点加入同一仓库结构。

## 第二部分：开发执行规范

### 1. 必读文档

在进行任何结构、设计、实现、测试、部署或依赖变更前，先阅读本文件。

如果任务执行过程中形成新的统一约束，例如新增目录约定、命名规则、测试流程、图片处理流程或部署要求，应先更新本文件，再继续实施。

`README.md` 面向项目所有者，用于 review 项目结构、依赖关系、构建流程和当前状态。修改技术结构、依赖或构建流程时，应同步更新 `README.md`。

### 2. 任务拆解

对于大任务、模糊要求或影响范围不明确的请求，禁止直接一次性执行。

执行前应先确认目标、边界和验收方式。必要时向用户提出简短澄清问题。确认后将任务拆解为多个小任务，每个小任务应有明确产出和可验证结果。

实施时按小任务分步推进，完成关键步骤后进行检查，例如类型检查、构建、测试、目录核对或人工可见结果说明。

### 3. 项目结构

当前仓库按静态个人网站 monorepo 规划：

```text
apps/
  photography/   # 当前实现重点
  home/          # 后续主页网站
  resume/        # 后续简历网站
```

`apps/photography` 的代码和资源应按以下规则放置：

```text
apps/photography/
  public/images/photography/  # 静态摄影图片
  src/
    components/               # 复用 UI 组件
    content/                  # Markdown 或内容文件
    data/                     # 摄影作品元数据
    styles/                   # 全局样式
    types/                    # 类型定义
```

第一版不引入后端服务、认证、数据库、访问统计或复杂服务端依赖。

### 4. 技术栈

摄影站使用 React + Vite + TypeScript + Tailwind CSS。动效使用 Framer Motion，但应保持克制，服务于图片浏览和内容展示。

第一版使用 npm 作为包管理器，并提交 `package-lock.json`。通过 `.nvmrc` 或 `.node-version` 固定 Node.js LTS 版本，并让 Cloudflare Pages 使用相同版本。

### 5. 命名与代码风格

组件使用 `PascalCase`，例如 `GalleryGrid.tsx`。Hooks、工具函数和普通函数使用 `camelCase`。

优先使用函数组件和 typed props。共享类型放在 `src/types/`。照片、分类、查看器等数据模型应保持类型化。

Tailwind class 保持局部、可读。只有当样式确实跨组件复用时，才抽成共享样式或组件变体。

组件应优先消费结构化数据，不在组件内硬编码摄影作品信息。

### 6. 摄影作品数据

摄影作品元数据放在 `apps/photography/src/data/`，内容型文本可放在 `apps/photography/src/content/`。

每张照片至少包含：

```text
src
alt
width
height
category
title
year
```

可选字段包括：

```text
location
featured
order
slug
```

新增作品时优先更新图片文件和元数据文件，不应为单张作品修改组件结构。

### 7. 图片与资产

不要提交未经压缩的原图。优先使用 WebP 或 AVIF，设置固定宽高或 `aspect-ratio`，并对非首屏图片使用 lazy loading。

如果图片存放在 `public/images/photography/`，不要假设 Vite 会在构建时自动压缩这些资源；应在提交前完成压缩，或提供明确的 `npm run optimize:images` 脚本。

如果后续改用 `src/assets/`，可以再考虑接入 Vite 图片处理插件，但第一版不强制。

所有可见摄影作品必须提供有意义的 `alt` 文本，不使用空泛描述。

### 8. 可访问性与 SEO

图片查看器必须支持键盘关闭、焦点管理和移动端触控操作。

页面动效需要尊重用户的 reduced-motion 偏好。

页面应包含基础 SEO 信息，包括标题、描述、favicon 和 Open Graph 图片。

移动端体验必须重点检查，尤其是图库布局、分类导航、图片查看器和触控关闭行为。

### 9. 测试与质量门槛

第一版最低质量门槛是：

```bash
npm run typecheck
npm run build
```

`npm run test` 只有在 Vitest 已配置后才要求通过。

推荐后续为数据工具和组件行为添加 Vitest，重点覆盖照片过滤、空状态和元数据校验。Playwright 可后续用于浏览器 smoke test，例如首页加载、图库渲染、分类筛选、图片查看器打开和关闭。

测试文件应靠近被测试代码命名，例如 `GalleryGrid.test.tsx` 或 `photos.test.ts`。

### 10. 开发命令

当前根目录已提供摄影站 workspace 脚本，优先从仓库根目录运行：

```bash
npm install
npm run dev:photography
npm run validate:photos
npm run typecheck:photography
npm run build:photography
```

`npm run validate:photos` 用于检查 `apps/photography/src/data/photos.json` 是否包含必填字段、数值尺寸和唯一 slug。修改照片元数据后应优先运行该命令。

也可以从 `apps/photography` 目录运行应用内命令：

```bash
npm install
npm run dev
npm run validate:photos
npm run typecheck
npm run build
```

配置 Vitest 后可运行：

```bash
npm run test
```

### 11. 部署

Cloudflare Pages 当前部署摄影站：

```text
Root directory: apps/photography
Build command: npm run build
Build output directory: dist
Framework preset: Vite
```

后续 `home`、`photography`、`resume` 可作为三个 Cloudflare Pages 项目独立部署、独立绑定子域名。

### 12. 安全

不要提交密钥、服务器凭据、手机号、身份证号、政府 ID 或其他敏感信息。

简历站后续如提供 PDF 下载，应避免包含不希望公开的个人隐私信息。

第一版不接入访问统计，避免额外数据收集。

### 13. 提交与 PR

提交信息使用简洁祈使句，例如 `Add photography gallery layout` 或 `Validate photo metadata`。

PR 应说明可见变化、列出已运行的测试或构建命令，并在涉及 UI 变化时附截图。
